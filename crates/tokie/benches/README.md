## Tokenizer Loading Time Comparison

Comparison of tokie's `.tkz` binary format vs HuggingFace tokenizers JSON loading.

```
cargo bench --bench loading
```

| Model | HF tokenizers (JSON) | tokie (.tkz) | Speedup |
|-------|---------------------|--------------|---------|
| bert | 10.19 ms (0.4 MB) | 2.62 ms (1.7 MB) | **3.9x** |
| gpt2 | 31.78 ms (1.3 MB) | 4.77 ms (3.1 MB) | **6.7x** |
| llama3 | 133.70 ms (8.7 MB) | 14.89 ms (8.4 MB) | **9.0x** |
| cl100k | 64.94 ms (4.0 MB) | 10.33 ms (6.6 MB) | **6.3x** |
| o200k | 152.62 ms (9.3 MB) | 20.94 ms (13.1 MB) | **7.3x** |

**tokie's .tkz format is 3.9x - 9.0x faster** than HuggingFace tokenizers loading from JSON.

The `.tkz` files are larger because they include pre-built DAAC (Double-Array Aho-Corasick) automaton state, but loading skips JSON parsing and automaton building entirely, achieving ~600 MB/s throughput.

### Setup for Loading Benchmarks

The loading benchmarks require tokenizer files in specific locations:

1. **tokie .tkz files** in `models/`:
   ```bash
   # These are generated by the regenerate_models example
   cargo run --release --example regenerate_models
   ```

2. **HuggingFace JSON files** in `/tmp/tokenizers/`:
   ```bash
   mkdir -p /tmp/tokenizers
   curl -sL "https://huggingface.co/bert-base-uncased/raw/main/tokenizer.json" -o /tmp/tokenizers/bert_tokenizer.json
   curl -sL "https://huggingface.co/openai-community/gpt2/raw/main/tokenizer.json" -o /tmp/tokenizers/gpt2.json
   curl -sL "https://huggingface.co/Xenova/gpt-4/raw/main/tokenizer.json" -o /tmp/tokenizers/cl100k.json
   curl -sL "https://huggingface.co/Xenova/gpt-4o/raw/main/tokenizer.json" -o /tmp/tokenizers/o200k.json
   # llama3 requires authentication
   curl -sL -H "Authorization: Bearer $HF_TOKEN" \
     "https://huggingface.co/meta-llama/Llama-3.2-1B/raw/main/tokenizer.json" \
     -o /tmp/tokenizers/llama3.json
   ```

## Algorithm Details

### HeapBytePairEncoder
- Uses binary heap (min-heap) for merge priority queue
- Arena-based linked list for token storage
- Lazy deletion with generation tracking
- Correctly handles non-topological merge orders (e.g., LLaMA 3)
- O(n log n) complexity

### BacktrackingBytePairEncoder
- Uses Aho-Corasick automaton for greedy longest-match
- Backtracking when adjacent tokens form invalid pairs
- Optimized for tiktoken-style tokenizers where token_id == merge_rank
- O(n) complexity with small constant factor
- **Best for**: GPT-2, GPT-3.5, GPT-4 (cl100k), and other tiktoken-style tokenizers

## Correctness Notes

The two encoders are designed for different tokenizer types:

1. **BacktrackingBytePairEncoder**: For tiktoken-style tokenizers (GPT-2, GPT-4) where token_id correlates with merge priority. Produces 100% identical output to HuggingFace tokenizers.

2. **HeapBytePairEncoder**: For tokenizers with non-topological merge orders (LLaMA 3). Simulates exact BPE merge process. Matches HuggingFace tokenizers exactly (8/8 test cases).

## Parallel Encoding Benchmark

Comparison of encoder throughput with parallel pretokenization + encoding using `std::thread::scope` and `memchunk::chunk`. Each thread handles both pretokenization and encoding for its chunk.

**Test Environment:**
- Text: War and Peace (3.36 MB)
- Hardware: Apple Silicon M3 Pro (8 performance cores)

### Sequential (Single-Threaded)

| Encoder | Time | Throughput | Tokens |
|---------|-----:|-----------:|-------:|
| BytePairEncoder (O(n²)) | 86.3 ms | 38.9 MB/s | 921,399 |
| HeapBytePairEncoder (O(n log n)) | 219.6 ms | 15.3 MB/s | 921,399 |
| BacktrackingBytePairEncoder (O(n)) | 53.9 ms | 62.3 MB/s | 921,399 |

### Parallel (8 Threads)

| Encoder | Time | Throughput | Speedup |
|---------|-----:|-----------:|--------:|
| BytePairEncoder (O(n²)) | 20.7 ms | 162.1 MB/s | 4.2x |
| HeapBytePairEncoder (O(n log n)) | 44.5 ms | 75.5 MB/s | 4.9x |
| BacktrackingBytePairEncoder (O(n)) | 11.3 ms | 298.0 MB/s | 4.8x |

### Key Findings

1. **Correctness**: All three encoders produce identical output (921,399 tokens)
2. **Simple O(n²) wins for small inputs**: With pretokenization splitting text into small pieces (avg ~4.3 bytes for GPT-2), the simple scan-and-merge outperforms the heap-based O(n log n) algorithm
3. **Near-linear speedup**: All encoders achieve ~4-5x speedup with 8 threads
4. **Backtracking is fastest**: For tiktoken-style tokenizers where token_id == merge_rank, the backtracking encoder achieves 298 MB/s parallel throughput

## SentencePieceBPE Benchmark

Comparison of SentencePieceBPE (Radix Heap + memchunk parallel chunking) against HuggingFace tokenizers for SentencePiece-style models.

**Test Environment:**
- Model: Mistral-7B-v0.1 (32k vocab, ~59k merges)
- Text: War and Peace (3.36 MB)
- Hardware: Apple Silicon (8 CPUs)

```
cargo run --release --example verify_sentencepiece_parallel
```

### Performance

| Implementation | Tokens | Time | Throughput |
|----------------|-------:|-----:|-----------:|
| HuggingFace tokenizers | 912,333 | 1,017 ms | 3.3 MB/s |
| SentencePieceBPE (sequential) | 912,333 | 636 ms | 5.3 MB/s |
| SentencePieceBPE (parallel) | 912,333 | 132 ms | 25.5 MB/s |

### Speedup

- **Sequential vs HuggingFace**: 1.6x faster
- **Parallel vs HuggingFace**: **7.7x faster**
- **Parallel vs Sequential**: 4.8x speedup (8 threads)

### Correctness

All implementations produce **identical output** (912,333 tokens):
- Sequential vs HuggingFace: MATCH
- Parallel vs HuggingFace: MATCH
- Parallel vs Sequential: MATCH

### Implementation Details

SentencePieceBPE uses:
- **Radix Heap**: O(1) amortized priority queue operations, exploiting BPE's monotonic rank property
- **memchunk**: SIMD-accelerated text chunking with `consecutive()` and `forward_fallback()` for correct metaspace boundary splitting
- **Scoped threads**: Each thread encodes its chunk independently, results are concatenated

The `consecutive()` option ensures chunks split at the START of metaspace runs (not mid-sequence like `▁▁▁`), and `forward_fallback()` searches forward when no boundary exists in the backward window, avoiding mid-word hard splits
